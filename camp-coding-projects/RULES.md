# PixelPad Kids' Workbook System — Design Rules

This document captures **every rule** we've agreed on while building this teaching system, so
anyone (or any future Claude session) can extend it consistently. It's the contract behind
`workbooks.py`.

---

## 1. What this project is

A printable, step-by-step **coding curriculum** for ~9-year-old kids, built on
[PixelPad](https://pixelpad.io) — a browser-based **Python 2D game engine** (PIXI.js +
Skulpt) where a whole game is one standalone HTML file.

For each game we generate **two** files plus a shared index:

| File | Purpose |
|------|---------|
| `<slug>-workbook.html` | The **printable workbook** — screenshots of the editor with the exact lines to type, highlighted green, one small step at a time. Has a **Print to PDF** button. |
| `<slug>-final.html` | The **finished, playable game** — the same code assembled and running, with the IDE chrome hidden. |
| `index.html` | A hub linking every workbook and finished game. |

Everything is generated by **`workbooks.py`** — never hand-edit the HTML. Change the rules or
game definitions in the Python and regenerate (`python workbooks.py`).

---

## 2. The kid audience comes first

The whole point is that a 9-year-old can type each step and **understand** it. Every rule below
exists to remove confusion and keep momentum.

- **Kids hate reading**, so instructions are *short* — one friendly sentence that teaches the
  concept, often ending in an encouraging "Press Play - ...!".
- **Frequent wins.** Steps are ordered so the child can press **Play** and *see a change* as
  early and as often as possible (e.g. in Brick Breaker you press Play after the first row of
  bricks, then again after the second).

---

## 3. Code-style rules (what the kids type)

These are enforced automatically by `workbooks.py` so we never break them by accident.

1. **No arrays / lists.** Use individually named objects (`Game.spikeA`, `Game.spikeB`), not a
   list. Kids don't meet indexing yet.
2. **Kid-friendly math only.** No `abs()`, no modulo `%`, no division `/`. Use plain
   `+ - *` and simple comparisons (`< >`).
3. **Descriptive names**, never cryptic ones (`jumpStrength`, `gameSpeed`, `clickPower`).
4. **Every variable is prefixed** — `self.` inside a class, `Game.` inside a room. A child
   never sees a lonely bare name like `score = 0`; they see `self.score = 0` or
   `Game.score = 0`. *(Handled by `prefix_locals()`. Text inside quotes is never touched, so
   sprite names like `'bird.png'` stay intact.)*
5. **`if` is never on one line with the code it runs.** `if x: y` is always split into
   ```python
   if x:
       y
   ```
   so the structure is visible. *(Handled by `split_if()`.)*
6. **At most 6 new lines per step.** A step shows only enough code to produce one testable
   change, so the kid can Play and check it. Longer blocks are split into "Step N" /
   "Step N (keep going)" pages, and we only ever split at an `if`-block boundary so an `if`
   never gets separated from its body. *(Handled by `paginate()` + `chunk_units()`, `maxn=6`.
   A single indivisible `if`-block may exceed 6 — that's allowed.)*
7. **Label text is white** — `text()` objects set `.color = "white"` (also `.halign`,
   `.fontSize` where needed) so they read on any background.

---

## 4. Sprite rules (the pixel art)

- **Native-size sprites.** Each texture is generated at the **exact pixel dimensions** the kid
  should draw (e.g. `64×64`, `120×20`). The typed code contains **no `scaleX`/`scaleY`
  multipliers** — the sprite is drawn at its real size. *(Constant `self.scaleX = <int>` lines
  are read for the texture size by `sprite_dims()` and then stripped from the displayed code by
  `strip_scale()`.)*
- **The setup page lists every sprite** with its dimensions and colour, e.g.
  *"player.png — draw it 38×38, orange"*, so the child makes the art the right size first.
- **A sprite name may not be used at two different sizes** (the generator raises an error if so —
  e.g. Pong's paddle and ball had to become `paddle.png` + `ball.png`).
- **Dynamic scaling is allowed for effects.** Only *constant integer* scale lines are stripped;
  a variable scale such as `self.scaleX = Game.blockW` (Stack Tower's shrinking block) or a
  pop animation `self.scaleX = 1 + self.pop` (Cookie Clicker) is kept and runs normally.

## 5. Engine / game-structure rules

- **`start()` runs once, `loop()` runs every frame.** State that must persist lives on `self.`
  (per object) or `Game.` (global). `Game` is the persistent root and its `loop` runs every
  frame regardless of the active room.
- **Coordinates:** `(0,0)` is the **centre**; **`y+` is UP**. Angles in degrees, positive = CCW.
- **Collision** is by bounding box: `get_collision(self, 'Class')` returns the object or
  `False`.
- **Rooms = levels/screens.** Changing room destroys non-persistent objects. **A level is just
  a room** — to add a level you add another room (see Platformer's `Level1` / `Level2`), point
  the previous level's `Game.nextLevel` at it, and `set_room(Game.nextLevel)` on reaching the goal.
- **Never call `set_room()` from a dying object's own `loop`** — it crashes the engine. Instead
  set a `Game.dead = True` flag and do the `set_room('GameOver')` from `Game.loop` (proven safe).
- **Lives + a real GameOver room.** Where it makes sense a game tracks `Game.lives` and, on
  losing, switches to a separate **`GameOver` room** (not just on-screen text). Tapping there
  resets state and `set_room('Play')` (or `Level1`). Timer games (Whack-a-Mole, Cookie Clicker)
  use a countdown instead of lives but still get the GameOver room.
- **Every GameOver screen shows the final score** under the big message — *"Final Score: N"*,
  or the themed variant (Pong: *"You 5   Cpu 2"*, Cookie Clicker: *"Cookies: N"*). *(Handled by
  the shared `SCORELINE()` helper and the `GO()` game-over trio.)*
- **Fairness matters.** Hazards are spaced so the game is winnable — e.g. Geometry Dash keeps
  its spikes a fixed distance apart by looping a recycled spike forward by the full loop length
  (`number_of_spikes × gap`), giving rhythmic, always-clearable spacing instead of random clumps.

## 6. Game-selection rules

- **20 games**, each **functionally distinct** — no two play the same way.
- **Popular, recognisable** games kids actually play.
- Each must teach a concept a 9-year-old can grasp with the code-style rules above.

Current line-up: Catch the Fruit, Whack-a-Mole, Flappy Bird, Subway Surfers, Geometry Dash,
Crossy Road, Pong, Brick Breaker, Doodle Jump, Space Shooter, Helicopter, Fruit Slice,
Asteroid Dodge, Stack Tower, Fishing, Rhythm Tap, Lander, Platformer, Cookie Clicker, Pac-Man.

## 7. Workbook layout & printing rules

- **One editor screenshot per step**, mimicking PixelPad's IDE in **light mode** (no dark mode),
  using PixelPad's colours.
- **Sidebar shows Classes and Rooms only** — no sprite list (rooms shown only if the game has
  them). The current class/room is highlighted.
- **Only one tab is shown** — *Start* **or** *Loop*, whichever the step is for — labelled like
  PixelPad (e.g. **"Player Start"**), **centred and bold**.
- **Only the new lines are shown**, highlighted **green**, numbered at the **correct cumulative
  line number** based on everything typed in earlier steps.
- **A `PLAY` badge** marks steps where the child can press Play and see progress.
- **Pack steps densely** — multiple steps per page when they fit (never wastefully one-per-page).
- **Text must never clip.** Long code lines **wrap** with a hanging indent (the line number stays
  in its own column); the editor is `max-width:100%` so it always fits the printable Letter page
  (≈710px inside 14mm margins). Verified at both screen and print widths.
- **Print to PDF button** is fixed top-right and hidden when printing (`@media print`).

---

## 8. How `workbooks.py` is organised

- `GAMES` — a list of game dicts: `title`, `slug`, `classes`, `rooms`, `textures`, and `steps`.
- `S(asset, tab, instr, lines, play)` — one authored step (which class/room, Start or Loop,
  the instruction, the raw lines, whether a Play badge shows).
- `GO()` — the standard game-over trio (the `Game.loop` dead→GameOver handler, the GameOver
  message, and the tap-to-restart loop). Now includes `SCORELINE()`.
- `SCORELINE(expr, instr)` — a GameOver step that prints the final score.
- Helpers that enforce the rules: `sprite_dims`, `strip_scale`, `split_if`, `prefix_locals`,
  `chunk_units`, `paginate`, `prep`.
- `assemble(game)` builds the playable `-final.html`; `render(game)` builds the printable
  `-workbook.html`; `setup_block(game)` writes the "Before you start" sprite/class/room list.
- Colours come from `RGB`; `solid_png()` generates each native-size texture as a data URI.

**To add a game:** append a dict to `GAMES` following the rules above, then run
`python workbooks.py`. **To change a rule:** edit the relevant helper so it applies to all 20
games at once.

---

## 9. Verification rule (don't skip)

Before claiming any change works, **test it in a real headless browser**: load the `-final`
file, drive the engine frame-by-frame (`ppApp._ticker.update(...)` in one monotonic loop),
simulate input, and read live state via `Sk.globals.Game`. For the workbooks, also check that
**no `.src` line overflows** its box at print width. WebGL screenshots time out — assert on
state, not pixels.
